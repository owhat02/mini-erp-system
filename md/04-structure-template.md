# Spring Boot BE 구조 설계 요청서 템플릿

> 목적: **구조만 생성**하기 위한 요청서 템플릿  
> 주의: **내용을 채우지 말고, 항목/섹션 구조만 유지**  
> 기준: 설계도 MD를 Single Source of Truth로 사용

---

## 프롬프트 본문

너는 10년차 백엔드 리드이자 부트캠프 팀프로젝트 감리자다.  
목표는 **“12일 내 구현 가능한” Spring Boot 패키지 구조를 생성**하는 것이다.  
**과설계 금지, Git 충돌 최소화가 최우선**이다.

### 중요
- 아래 제공되는 **[설계도 MD]**가 유일한 기준(Single Source of Truth)이다.
- 추가 질문 없이, 설계도 MD를 읽고 필요한 정보를 스스로 추출해라.
- 출력은 **패키지 구조 생성 + 혼돈 방지 규칙 + 감리자 중간 검토 보고서 + 다음 단계 + 분업 제안**까지 포함한다.
- **출력 이후에는 어떤 걸 추가했는지 파일 추가 내역도 작성**해라.
- **설명은 이론보다 실행 중심**으로 작성한다.
- **내용을 채워 넣지 말고, 구조만 생성**한다.
- 뭐 어떻데 추가 했는지 알려줘

## (B) BE 패키지 트리
```text
root/
 ├── global/
 │    ├── config/
 │    ├── exception/
 │    └── response/
 └── domain/
      ├── ...
      │    ├── controller/
      │    ├── service/
      │    ├── repository/
      │    ├── entity/
      │    └── dto/
      └── ...
```

조건:
- domain 기반 패키지 구조
- 각 도메인 내부:
  - controller/
  - service/
  - repository/
  - entity/
  - dto/
- global 패키지 포함:
  - global/config
  - global/exception
  - global/response
- 인증 방식에 따라 security 또는 auth를 일관성 있게 구성
- Controller → Service → Repository 흐름이 구조에 반영되어야 함

## (C) 패키지/레이어 혼돈 방지 규칙 (필수 12개)

각 항목은 아래 형식으로만 작성:
- ❌ 금지
- ✅ 대안
- 💡 예시

반드시 포함할 항목:
1. Controller에 비즈니스 로직 작성
2. Entity를 그대로 API 응답으로 반환
3. DTO 패키지가 도메인 밖에 흩어짐
4. Service에서 DB 접근 로직 직접 구현
5. 예외/응답 포맷이 도메인마다 다름
