# [상세 도메인 설계서 v5.0] 통합 그룹웨어 시스템 (SmartWork)

---

## 1. 프로젝트 개요 (Project Overview)

### 1.1 프로젝트 목적
- **통합 관리 환경 구축**: 사내 프로젝트 업무 트래킹과 인사 행정(근태/휴가)을 단일 플랫폼으로 통합하여 데이터 파편화 및 관리 효율 저하 문제를 원천 해결함.
- **업무 프로세스 최적화**: 실시간 업무 상태 공유와 자동화된 연차 승인/차감 로직을 통해 불필요한 행정 소요 시간을 단축하고 본연의 업무에 집중할 수 있는 환경을 제공함.
- **데이터 정합성 확보**: 프로젝트 진행도와 인사 데이터를 실시간 동기화하여 신뢰할 수 있는 전사 관리 지표를 확보함.

### 1.2 핵심 가치 (Core Values)
- **통합(Integration)**: 분산된 업무 및 근태 데이터를 하나의 플랫폼으로 집중시켜 운영 복잡성 최소화.
- **투명(Transparency)**: 업무 진행도 시각화 및 결재 프로세스 공개를 통한 조직 내 신뢰도 향상.
- **정합(Consistency)**: 승인-차감 트랜잭션 보장을 통한 인사 데이터의 무결성 확보.
- **민첩(Agility)**: 실시간 알림 및 칸반 기반 트래킹을 통한 빠른 의사결정 지원.

### 1.3 프로젝트 범위 (Scope)
- **인증 및 보안**: JWT Access Token 기반 보안 설정, ROLE(관리자/사용자)별 접근 제어, 회원가입 유효성 검사.
- **업무 관리(PMS)**: 프로젝트 생성, Task 할당(매핑), 칸반 보드 기반 트래킹 및 진행도 시각화.
- **근태 및 인사(HRM)**: 출퇴근 기록, 직급별 연차 부여 기준 설정, 잔여 연차 자동 계산 로직.
- **전자 결재(e-Approval)**: 휴가 신청 기안, 관리자 승인/반려 프로세스 및 반려 사유 피드백 시스템.
- **시각화 서비스**: 통합 캘린더, 근태/프로젝트 대시보드 차트 및 실시간 알림.

### 1.4 주요 이해관계자
| 구분 | 역할 | 주요 관심사 |
|---|---|---|
| 일반 사원 | 시스템 사용자 | 본인 업무 상태 업데이트, 간편한 연차 신청 및 잔여일 실시간 확인 |
| 관리자 | 시스템 운영자 | 전사 프로젝트 진척도 모니터링, 팀원 근태 관리 및 신속한 결재 처리 |
| IT 관리자 | 시스템 관리자 | 데이터 정합성 유지, 권한 보안(JWT), 서버 및 데이터베이스 안정성 확보 |

## 2. 비즈니스 도메인 분석

### 2.1 핵심 비즈니스 프로세스

#### 2.1.1 사용자 가입 및 권한 승인 프로세스
`A[신규 방문자] ➔ B[회원가입 신청] ➔ C[관리자 승인 대기] ➔ D[관리자의 승인 처리] ➔ E[계정 활성화 및 ROLE 부여] ➔ F[로그인 후 서비스 이용]`

**상세 플로우**
- **가입 신청 단계**: 사용자가 ID, PW, 이름, 이메일, 직책 정보를 입력하여 가입을 신청함. 초기 상태는 '승인 대기'임.
- **관리자 승인 단계**: 관리자 전용 페이지에서 가입 신청 목록을 확인하고, 조직 구성원 확인 후 '승인' 또는 '거절'을 처리함.
- **권한 부여 단계**: 승인 시 해당 사용자에게 일반 사원(ROLE_USER) 또는 관리자(ROLE_ADMIN) 역할을 부여함.

#### 2.1.2 프로젝트 업무 및 태스크 관리 프로세스
`A[관리자의 프로젝트 생성] ➔ B[업무(Task) 등록 및 담당자 배정] ➔ C[사원의 업무 확인] ➔ D[업무 상태 변경(TODO➔DONE)] ➔ E[진척도 시각화 반영]`

**상세 플로우**
- **업무 배정 단계**: 상사가 업무 지시 내용(제목, 내용, 기간)을 등록하고 담당자를 매핑함.
- **업무 수행 단계**: 사원은 본인에게 배정된 프로젝트만 조회하여 업무를 진행하고 상태를 업데이트함.
- **모니터링 단계**: 업데이트된 상태는 대시보드 진행도 차트에 실시간으로 반영됨.

#### 2.1.3 연차 신청 및 결재 프로세스
`A[잔여 연차 확인] ➔ B[연차 신청서 작성] ➔ C[결재 대기(Pending)] ➔ D[관리자 검토 및 승인/반려] ➔ E[연차 차감 및 캘린더 반영]`

**상세 플로우**
- **기안 단계**: 사원이 신청 사유와 기간을 입력하여 결재를 요청함. 이때 시스템은 잔여 연차 일수와 신청 기간 내 주말/공휴일 포함 여부를 체크함.
- **결재 처리 단계**: 관리자가 내용을 검토 후 승인하면 상태가 APPROVED로 변경됨. 반려 시에는 반려 사유를 작성함.
- **후속 처리 단계**: 승인 즉시 연차 잔여 일수가 자동 차감되며, 통합 캘린더에 휴가 일정이 표시됨.

### 2.2 비즈니스 이벤트 (Business Events)
| 이벤트 | 트리거 (Trigger) | 결과 (Result) |
|---|---|---|
| 회원가입 신청 | 사용자가 가입 정보 제출 | 사용자 계정 '승인 대기' 상태 생성, 관리자 알림 |
| 출근 기록 | 사원이 '출근' 버튼 클릭 | clock_in_time 기록 및 지각 여부 판별 |
| 연차신청 | 사원이 연차신청서 작성 후 '기안'버튼 클릭 | 결재상태 'Pending(대기)' 생성, 관리자에게 결재 요청 알림 전송 |
| 연차 승인 | 관리자가 결재 승인 버튼 클릭 | remaining_annual_leave 차감 및 결재 상태 변경 |
| 업무 완료 | 사원이 Task 상태를 DONE으로 변경 | 프로젝트 전체 진척도 시각화 업데이트 |
| 연차 반려 | 관리자가 반려 사유 입력 후 거절 | 기안자에게 반려 사유 전달 및 상태 변경 |

### 2.3 비즈니스 규칙 (Business Rules)
- **계정 관련**: 신규 사용자는 관리자의 승인 전까지 서비스 이용이 불가함.
- **업무 관련**: 사원은 오직 본인이 배정받은(task_assignments) 프로젝트만 조회 및 수정할 수 있음.
- **연차 관련**: 잔여 연차 일수보다 많은 일수의 휴가는 신청할 수 없으며, 승인 시에만 연차가 차감됨.
- **연차 관련**: 연차 신청 시 UI에서 주말/공휴일 날짜 선택을 차단하고, 서버에서도 신청 기간에 주말/공휴일 포함 여부를 검증해야 함.
- **근태 관련**: 주말/공휴일 출근 기록은 특근 신청서 승인 사용자만 허용하며, 미승인 사용자의 출근 기록은 차단함.
- **보안 관련**: 모든 API 요청은 JWT Access Token 기반 인증이 필수이며(Access Token only), 직급 및 역할에 따라 접근 가능한 메뉴가 제한됨.

## 3. 핵심 도메인 객체 (Minimum Viable Product)

### 3.1 도메인 객체 식별 매트릭스
| 도메인 객체 | 유형 | 중요도 | 복잡도 | 비고 (Description) |
|---|---|---|---|---|
| User (사용자) | Entity | 높음 | 중간 | 시스템 이용 주체. 직급(Position)에 따른 연차 부여 기준 및 시스템 접근 권한(Role)을 관리함. |
| Project (프로젝트) | Entity | 높음 | 중간 | Task의 상위 업무 단위. 사용자에게 할당된 프로젝트 범위만 조회 대상으로 사용됨. |
| Task (업무) | Entity | 높음 | 높음 | 프로젝트의 최소 단위 업무. Task_No로 분류되며 시작/종료일 기반으로 진행 상태를 트래킹함. |
| TaskAssignment | Entity | 높음 | 중간 | 업무와 담당자 간 다중 배정(N:M) 매핑 객체. USER는 본인이 배정된 Task/프로젝트 데이터만 조회 가능함. |
| Approval (결재) | Entity | 높음 | 높음 | 연차/휴가 신청의 핵심 객체. 기안자와 승인자 간의 워크플로우를 관리하며 반려 사유(Reject_Reason)를 포함함. |
| Attendance (근태) | Entity | 중간 | 중간 | 일일 출퇴근 기록(Clock_in/out) 객체. 출근 일수를 산출하고 지각/결근 등 근태 상태를 판별함. |
| Role (권한) | Value Object | 중간 | 낮음 | 사용자 권한 구분(일반 사원, 관리자). 서비스 메뉴 및 결재 승인 권한 범위를 결정함. |
| TaskStatus (업무상태) | Value Object | 낮음 | 낮음 | 업무 진행 단계(TODO, DOING, DONE)를 관리하며 대시보드 시각화의 기준 데이터가 됨. |
| ApprovalStatus | Value Object | 낮음 | 낮음 | 전자결재의 현재 상태(대기, 승인, 반려)를 나타내며 User 연차 차감 로직의 트리거 역할을 함. |

### 3.2 상세 도메인 객체 정의

#### 3.2.1 User (사용자)
- **역할**: 시스템의 주체이며, 직급 및 역할(사원/관리자)에 따라 접근 권한이 달라짐.
- **주요 속성**
  - user_id: 사용자 고유 식별자 (PK)
  - user_name: 사용자 실명
  - user_email: 로그인 이메일 (Unique)
  - user_pw: 암호화된 비밀번호
  - position_name: 직책 (사원, 대리, 과장 등)
  - assign_role: 프로젝트 내 역할 (PM, Member)
  - user_role: 시스템 권한 (ROLE_USER, ROLE_ADMIN)
  - total_annual_leave: 총 연차
  - used_annual_leave: 사용 연차
  - remaining_annual_leave: 잔여 연차
- **주요 행동 (메서드)**
  - updateProfile(): 사용자 정보 수정
  - verifyPassword(): 로그인 시 비밀번호 일치 확인
  - changeRole(): 관리자가 사용자의 시스템 권한을 변경
- **비즈니스 규칙**
  - user_email과 user_id는 시스템 내에서 중복될 수 없음.
  - 관리자(ROLE_ADMIN) 권한을 가진 사용자만 관리자 페이지에 접근 가능.

#### 3.2.2 Task (업무)
- **역할**: 프로젝트를 구성하는 최소 단위의 업무이며, 진행 상태를 추적함.
- **주요 속성**
  - id: 업무 고유 식별자 (PK)
  - task_title: 업무 제목
  - task_content: 업무 상세 내용
  - start_date: 업무 시작 일자
  - end_date: 업무 마감 일자
  - task_state: 진행 상태 (TODO, DOING, DONE)
  - task_no: 프로젝트 분류 번호
- **주요 행동 (메서드)**
  - updateStatus(newState): 업무 상태를 변경 (예: TODO ➔ DOING)
  - assignUsers(users): TaskAssignment를 통해 다중 담당자를 배정
  - isOverdue(): 마감 기한이 지났는지 확인
- **비즈니스 규칙**
  - start_date는 end_date보다 이전 날짜여야 함.
  - 모든 Task는 반드시 하나 이상의 담당자(task_assignments)가 존재해야 함.

#### 3.2.3 Approval (전자결재)
- **역할**: 연차 및 휴가 신청에 대한 승인 프로세스를 관리함.
- **주요 속성**
  - app_id: 결재 고유 식별자 (PK)
  - requester_id: 기안자 ID (User FK)
  - approver_id: 승인자 ID (User FK)
  - app_type: 결재 유형 (ANNUAL, HALF, SICK)
  - used_days: 서버 계산된 연차 사용 일수
  - app_status: 결재 상태 (PENDING, APPROVED, REJECTED)
  - reject_reason: 반려 시 사유
  - created_at: 기안 일시
- **주요 행동 (메서드)**
  - approve(): 관리자가 결재를 승인 처리
  - reject(reason): 관리자가 사유를 입력하며 반려 처리
  - calculateDays(): 시작일/종료일을 기반으로 서버에서 사용 일수 자동 계산
- **비즈니스 규칙**
  - app_status가 APPROVED로 변경될 때만 실제 연차 잔여량이 차감됨.
  - 연차 신청 기간에 주말/공휴일이 포함되면 기안이 거부됨(서버 검증).
  - 본인이 본인의 결재를 승인할 수 없음(requester_id != approver_id).

#### 3.2.4 Attendance (근태)
- **역할**: 사용자의 출퇴근 기록을 관리하는 객체.
- **주요 속성**
  - att_id: 근태 고유 식별자 (PK)
  - user_id: 사용자 ID (User FK)
  - clock_in_time: 출근 시각
  - clock_out_time: 퇴근 시각
  - att_status: 근태 상태 (정상, 지각, 결근)
  - work_date: 근무 일자
- **주요 행동 (메서드)**
  - recordClockIn(): 출근 시각 기록 및 상태 판별
  - recordClockOut(): 퇴근 시각 기록
- **비즈니스 규칙**
  - 하루에 출근 기록(clock_in_time)은 한 번만 생성 가능.
  - 주말/공휴일 출근은 특근 신청 승인 사용자만 가능하며, 미승인 시 출근 기록이 거부됨.

### 4.1 개념적 관계도

### 4.2 관계 상세 설명
| 관계 | 카디널리티 | 설명 |
|---|---|---|
| User ↔ Task | N:M | 한 명의 사용자는 여러 업무를 담당할 수 있고, 한 업무에 여러 담당자가 지정될 수 있음 (TaskAssignment 사용) |
| Task ↔ TaskAssignment | 1:N | 하나의 Task는 하나 이상의 담당자 배정 정보를 가질 수 있음 |
| Project ↔ Task | 1:N | 하나의 프로젝트는 여러 개의 하위 업무(Task)로 구성됨 |
| User ↔ Approval | 1:N | 한 명의 사원은 여러 건의 연차/휴가 결재를 기안할 수 있음 |
| User ↔ Attendance | 1:N | 한 명의 사용자는 일자별로 여러 개의 근태 기록(출근/퇴근)을 가질 수 있음 |
| Project ↔ User | 1:N | 한 프로젝트에 여러 사원을 배치할 수 있음 |

## 5. 비즈니스 규칙 (Business Rules)

### 5.1 회원 및 권한 관련 규칙
| 규칙 ID | 규칙 내용 |
|---|---|
| BR-M001 | 신규 사용자는 관리자의 승인 전까지 'PENDING' 상태이며 로그인이 불가함 |
| BR-M002 | 사용자 ID와 이메일은 시스템 내에서 유일해야 함 |
| BR-M003 | '사원' 직급은 타인의 연차 결재를 승인할 수 없으며, 본인 관련 데이터만 조회 가능함 |

### 5.2 프로젝트 및 업무 관련 규칙
| 규칙 ID | 규칙 내용 |
|---|---|
| BR-A001 | 프로젝트 매니저(PM)만이 해당 프로젝트의 하위 Task를 생성하고 배정할 수 있음 |
| BR-A002 | 업무 상태가 'DONE'으로 변경된 경우에만 프로젝트 전체 진척도에 반영됨 |
| BR-A003 | 마감 기한(end_date)이 지난 업무는 대시보드에서 '지연' 상태로 강조 표시됨 |
| BR-A004 | Task 담당자 구조는 TaskAssignment 기반 다중 담당자만 사용하며, 단일 assignee 컬럼 방식은 사용하지 않음 |

### 5.3 연차 및 결재 관련 규칙
| 규칙 ID | 규칙 내용 |
|---|---|
| BR-L001 | 잔여 연차(remaining_days)가 신청 일수보다 적을 경우 결재 기안 자체가 불가함 |
| BR-L002 | 관리자가 결재를 'APPROVED' 처리하는 즉시 User의 연차 수량이 자동 차감됨 |
| BR-L003 | 연차 신청 기간에 주말/공휴일이 포함되면 신청이 거부되며, UI와 서버 모두에서 검증함 |

### 5.4 근태 및 특근 관련 규칙
| 규칙 ID | 규칙 내용 |
|---|---|
| BR-W001 | 주말/공휴일 출근 기록은 특근 신청서가 승인된 사용자만 허용함 |
| BR-W002 | 특근 미승인 상태에서 주말/공휴일 출근 시도 시 예외를 반환하고 기록을 저장하지 않음 |

## 8. 가정사항 및 제약조건

### 8.1 가정사항 (Assumptions)
- 사용자는 SmartWork 시스템을 도입한 기업의 임직원임을 가정함.
- 사용자는 기본적인 웹 브라우저(Chrome, Edge 등) 사용법에 익숙하며, 인터넷 환경이 확보됨.
- 모든 시간 기록 및 결재 데이터는 **한국 표준시(KST)**를 기준으로 함.
- 관리자 계정은 시스템 초기 구축 시 사전에 생성되어 배포됨을 가정함.

### 8.2 제약조건 (Constraints)
- **기술적 제약**: 백엔드는 Spring Boot, 프론트엔드는 React를 사용하여 개발하며, 데이터 무결성을 위해 RDBMS를 사용함.
- **개발 범위**: 음성/영상 통화 기능이나 고도화된 메신저 기능은 MVP 범위에서 제외하며, 프로젝트/연차 관리에 집중함.
- **보안 제약**: **JWT Access Token only** 기반 API 인증을 필수로 하며, 사용자 비밀번호는 반드시 단방향 암호화하여 저장함.
- **운영 제약**: 연차 계산 로직은 근로기준법 및 해당 기업의 내규를 우선하여 적용함.

## 10. 개발 전 필수 확정 8개

### 10.1 Task 담당자 구조
- **다중 담당자(`TaskAssignment`)로 확정**
- 단일 assignee 컬럼과 혼용 금지

### 10.2 USER 조회 범위
- USER는 업무 할당 시 **할당받은 업무/프로젝트 관련 데이터만 조회 가능**
- 할당받지 않은 프로젝트는 조회 불가

### 10.3 연차 데이터 위치
- 연차 데이터(총/사용/잔여)는 **User 테이블**에서 단일 관리

### 10.4 연차 신청 `usedDays` 계산 주체
- `usedDays`는 클라이언트 입력값을 신뢰하지 않고 **서버에서 계산**

### 10.5 인증 방식
- 인증은 **Access Token only**로 운영

### 10.6 승인-차감 트랜잭션 원칙
- 결재 승인 상태 변경과 User 연차 차감은 **동일 트랜잭션**으로 처리
- 둘 중 하나라도 실패하면 전체 롤백

### 10.7 연차 신청 주말/공휴일 검증
- UI: 주말/공휴일 날짜 선택 차단
- 서버: 신청 기간에 주말/공휴일 포함 여부 재검증(우회 요청 차단)

### 10.8 주말/공휴일 출근 예외 처리
- 주말/공휴일 출근은 특근 신청서 승인 사용자만 허용
- 미승인 사용자의 출근 요청은 예외 처리
